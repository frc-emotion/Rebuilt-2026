## 1. Swerve Drivetrain

### What it does
4-module swerve drive for omnidirectional movement. Field-centric control by default (robot moves relative to the field, not its own heading).

### Hardware
- **Motors**: 8x TalonFX (Kraken X60) — 4 drive + 4 steer
- **Encoders**: 4x CANcoder (one per module, for absolute steer position)
- **IMU**: Pigeon2 (gyro for field-centric heading + yaw rate for turret FF)
- **CAN Bus**: Default (rio) bus

### Code
- **Subsystem**: `CommandSwerveDrivetrain.java` extends CTRE's generated `TunerSwerveDrivetrain`
- **Constants**: `generated/TunerConstants.java` (auto-generated by Tuner X)
- **Control mode**: `SwerveRequest.FieldCentric` with open-loop voltage drive

### Key parameters
| Parameter | Value | Source |
|---|---|---|
| Max speed | `kSpeedAt12Volts` × 1.0 | TunerConstants |
| Max angular rate | 0.75 rot/s → rad/s | RobotContainer |
| Drive deadband | 10% of max speed | RobotContainer |
| Rotation deadband | 10% of max angular rate | RobotContainer |

### PID
Drive and steer PID are configured in `TunerConstants.java` by the Tuner X generator. These are already tuned for the physical robot's swerve modules.

### Associated commands
- **Default**: Field-centric drive from driver joystick (left stick = translate, right stick = rotate)
- `joystick.a()` → Brake (X-pattern wheel lock)
- `joystick.leftBumper()` → Seed field-centric heading
- `joystick.y()` → Seed pose facing red hub at ~4m
- D-pad → SysId characterization routines (temporary)

## 2. Turret

### What it does
Rotates the turret assembly (which carries the hood + shooter + camera) to aim at the hub. Uses MotionMagic for smooth position control with acceleration/deceleration profiling.

### Hardware
- **Motor**: 1x TalonFX (CAN ID 51) — Kraken X60
- **Encoder**: 1x CANcoder (CAN ID 53) — currently unused for feedback (rotor sensor used instead)
- **CAN Bus**: "mechanisms"

### Gear ratio
```
Motor rotor → Turret output
Ratio: 5.08:1 (5.08 motor rotations = 1 turret rotation)
```
This means:
- 1 turret rotation = 360 degrees of turret sweep
- The turret's position is reported in **mechanism rotations** (turret output shaft rotations)
- `SensorToMechanismRatio = 5.08` tells the TalonFX firmware to auto-convert rotor ticks to turret rotations

### Position units
All turret positions are in **mechanism rotations** (turret output rotations).
- `0.0` = straight forward (zeroed at boot)
- `+0.20` = forward soft limit (~72 degrees CW from front)
- `-0.58` = reverse soft limit (~209 degrees CCW from front)
- Total range: 0.78 rotations = **280.8 degrees** of sweep

### Soft limits
Both forward and reverse software limits are enabled in firmware. The motor will refuse to drive past these positions. This prevents the turret from crashing into the intake.

### Zero procedure
The turret is zeroed at boot (`turretMotor.setPosition(0)`). The robot must be powered on with the turret facing straight forward. If the turret is bumped, press operator right bumper to re-zero at current position.

### PID (Slot 0 — MotionMagic position control)

| Gain | Value | Purpose |
|---|---|---|
| kP | 20.0 | **Primary correction**. Units: volts per rotation of error. 20V/rot means 0.01 rot error (3.6 deg) produces 0.2V correction. This is aggressive — appropriate for a low-inertia turret that needs to track quickly. |
| kI | 5.0 | **Steady-state error elimination**. Accumulates error over time to push through static friction (cable drag, gear mesh friction). Units: volts per rotation-second. |
| kD | 0.0 | Not used. MotionMagic's trapezoidal profile already handles deceleration. |
| kS | 0.0 | Static friction compensation — not tuned yet. Should be set to the minimum voltage to start moving. |
| kV | 0.0 | Velocity feedforward — not needed for MotionMagic position control (the profile generator handles velocity). |
| kG | 0.0 | Gravity compensation — turret rotates horizontally, no gravity load. |

**MotionMagic profile constraints:**

| Parameter | Value | Meaning |
|---|---|---|
| CruiseVelocity | 1.0 RPS | Max turret rotation speed (360 deg/s) |
| Acceleration | 4.0 RPS² | Ramp rate — reaches cruise in 0.25s |
| Jerk | 40.0 | Smoothing — prevents jerky starts/stops |

**Tuning notes:**
- If turret oscillates around setpoint → decrease kP or increase kD
- If turret doesn't quite reach setpoint (steady-state error) → kI handles this
- If turret slams at endpoints → decrease CruiseVelocity or Acceleration
- Current limits: 80A stator, 40A supply — protect against stall

### Associated commands
- **Default**: `TurretAutoAimCommand` — auto-switches between MANUAL (joystick) and TRACKING (vision)
- **D-pad**: Preset positions (center/right/left/far-left) — interrupts default while held
- **Right bumper**: Re-zero turret at current position
- **ManualTurretCommand**: Emergency fallback — direct voltage from joystick (±3V)

## 3. Hood

### What it does
Adjusts the launch angle of the ball by tilting the hood mechanism. Small range of motion (0 to 0.08 rotations) but critical for scoring at different distances.

### Hardware
- **Motor**: 1x TalonFX (CAN ID 52)
- **Encoder**: 1x CANcoder (CAN ID 54)
- **CAN Bus**: "mechanisms"

### Gear ratio
```
Motor rotor → Hood output
Ratio: 155/12 = 12.917:1
```
This means:
- 12.917 motor rotations = 1 hood output rotation
- `SensorToMechanismRatio = 12.917` auto-converts rotor to hood rotations
- Hood total travel: 0.0 to 0.08 rotations = **28.8 degrees**

### Position units
All hood positions are in **mechanism rotations** (hood output rotations).
- `0.0` = flat/down (zeroed at boot, assumed to be at bottom hard stop)
- `0.08` = maximum tilt up

### Soft limits
Both forward (0.08) and reverse (0.0) software limits are enabled.

### Zero procedure
Hood is zeroed at boot (`hoodMotor.setPosition(0)`). The robot must be powered on with the hood at its bottom hard stop.

### PID (Slot 0 — MotionMagic position control)

| Gain | Value | Purpose |
|---|---|---|
| kP | 100.0 | Very aggressive because the total travel is tiny (0.08 rot). 0.001 rot error = 0.1V correction. Needed to hold position precisely against ball impact forces. |
| kI | 50.0 | High integral to push through gear friction and hold exact angle. |
| kD | 0.0 | Not used — MotionMagic profile handles deceleration. |
| kS/kV/kG | 0.0 | Not tuned. kG could help if the hood has a gravity load. |

**Why kP=100?** The hood's total range is only 0.08 rotations. A "large" error of 0.01 rot (12.5% of range) produces 1.0V — just enough to correct without being violent. For comparison, the turret's kP=20 operates over a 0.78 rot range, so 0.01 rot error there is only 1.3% of range.

**MotionMagic profile:**

| Parameter | Value |
|---|---|
| CruiseVelocity | 1.0 RPS |
| Acceleration | 2.0 RPS² |
| Jerk | 20.0 |

### Associated commands
- **Default**: Holds last commanded position via MotionMagic (set in RobotContainer)
- **X/Y/B buttons**: Preset hood angles (down/mid/up)
- **ShootCommand**: Sets hood angle from interpolation table based on distance
- **ManualHoodCommand**: Emergency fallback — direct voltage from joystick (±3V)

## 4. Shooter (Flywheel)

### What it does
Spins the flywheel to launch balls at the hub. Uses closed-loop velocity control to maintain consistent shot speed.

### Hardware
- **Motor**: 1x TalonFX (CAN ID 50)
- **CAN Bus**: "mechanisms"
- **No external encoder** — uses internal rotor sensor

### Gear ratio
No external gear reduction documented — the flywheel appears to be direct-drive or 1:1. `SensorToMechanismRatio` is not set in config (defaults to 1.0). All velocities are in **rotor RPS**.

### Velocity units
- Setpoints are in **RPS** (rotations per second of the motor/flywheel)
- Max: 400 RPS (clamped in `setShooterSpeed()`)
- Typical operating range: 45-74 RPS (from interpolation table)
- Tolerance: 0.5 RPS for `atShooterSetpoint()` check

### Voltage limits (NEW)
```
PeakForwardVoltage = 10.0V  — caps maximum spin-up voltage
PeakReverseVoltage = 0.0V   — NEVER applies reverse voltage
```
When the shooter is commanded to stop or slow down, the motor simply coasts. Friction and air resistance bring it to a stop naturally. This prevents the motor from fighting the flywheel's inertia (which wastes power and could damage belts).

### PID (Slot 0 — VelocityVoltage control)

| Gain | Value | Purpose |
|---|---|---|
| kS | 0.15 | **Static friction compensation**. Minimum voltage to overcome friction and start spinning. Units: volts. |
| kV | 0.12 | **Velocity feedforward**. Units: volts per RPS. At 60 RPS → 7.2V feedforward. This is the primary output — PID only handles the residual error. |
| kP | 0.3 | **Proportional correction**. Units: volts per RPS error. 1 RPS error → 0.3V extra. |
| kI | 0.0 | Not used — flywheel doesn't need integral (no static load to overcome). |
| kD | 0.0 | Not used — flywheel has high inertia, naturally dampens oscillation. |

**How VelocityVoltage works:**
```
output_volts = kS × sign(velocity) + kV × target_velocity + kP × velocity_error
```
At 60 RPS target:
- Feedforward: 0.15 + 0.12×60 = **7.35V**
- If actual is 59 RPS: error correction = 0.3×1 = **0.3V**
- Total: **7.65V** (under the 10V cap)

**Tuning notes:**
- If flywheel takes too long to spin up → increase kV or kP
- If flywheel oscillates around setpoint → decrease kP
- kV is the most important gain — it should produce ~90% of the required voltage at operating speed

### Associated commands
- **ShootCommand** (auto mode): Sets speed from interpolation table based on distance
- **ShootCommand** (manual mode): Fixed 40 RPS
- **CalibrationShootCommand**: Adjustable speed via NetworkTables
- **ManualShooterCommand**: Emergency fallback — speed from joystick trigger

## 5. Indexer

### What it does
Three-stage ball transport system that moves balls from intake to shooter:
1. **Horizontal** — moves balls laterally from intake into the vertical channel
2. **Vertical** — lifts balls upward toward the shooter
3. **Upward** — final feeder that pushes balls into the flywheel

### Hardware
- **Motors**: 3x TalonFX (CAN IDs 31, 32, 33)
- **CAN Bus**: "mechanisms"
- No external encoders — velocity control only

### Velocity units
All three motors use `VelocityVoltage` control. Speeds are in **RPS** (motor rotor rotations per second).
- Default speed: 50 RPS for all three stages
- Horizontal runs negative (-50 RPS) due to mounting direction

### PID (all three motors identical — Slot 0)

| Gain | Value | Purpose |
|---|---|---|
| kS | 0.15 | Static friction compensation |
| kV | 0.12 | Velocity feedforward (volts per RPS) |
| kP | 0.3 | Proportional correction |

These are simple velocity controllers — the indexers just need to spin at a consistent speed. The gains are identical to the shooter because they're the same motor type running a similar velocity control loop.

### Associated commands
- **ShootCommand**: Runs all three indexers when `isAimed()` is true
- **runIndexer**: Runs all three at default speeds (used in auto)
- **stopIndexer**: Stops all three (instant command)
- **Left trigger**: Runs vertical indexer only (for manual ball staging)
- **Right stick click**: Reverse all indexers at 50% speed (clear jams)

## 6. Intake

### What it does
Two-part mechanism:
1. **Pivot** — rotates the intake arm between stowed (in) and deployed (out) positions using MotionMagic
2. **Roller** — spins to grab balls when deployed

### Hardware
- **Pivot motor**: 1x TalonFX (CAN ID 20)
- **Roller motor**: 1x TalonFX (CAN ID 21)
- **Pivot encoder**: 1x CANcoder (CAN ID 22) — absolute position for the pivot
- **CAN Bus**: "mechanisms"

### Gear ratios and feedback
```
Motor rotor → CANcoder: 27:1 (RotorToSensorRatio = 27.0)
CANcoder → Pivot output: 1:1 (SensorToMechanismRatio = 1.0)
```
The CANcoder is mounted 1:1 with the pivot output shaft. The motor goes through a 27:1 reduction before reaching the CANcoder/pivot. The TalonFX uses the CANcoder as a **remote sensor** (`FeedbackSensorSource = RemoteCANcoder`), so all positions are in **pivot output rotations**.

### Position units
- `0.3 rot` = stowed (INTAKE_IN_ANGLE)
- `0.653 rot` = deployed (INTAKE_OUT_ANGLE)
- Soft limits: 0.25 to 0.70 rotations (margin on both sides)
- Total travel: ~0.353 rotations = **127 degrees**

### Roller velocity
- Speed: 55 RPS via VelocityVoltage
- Stops when intake is stowed

### PID — Pivot (Slot 0 — MotionMagic)

| Gain | Value | Purpose |
|---|---|---|
| kP | 13.0 | Position correction. Moderate — pivot has more travel than hood so doesn't need to be as aggressive. |
| kI | 0.0 | Not used. |
| kD | 0.0 | Not used. |
| kG | 0.0 | Not tuned. **Should be set** — the pivot arm has a gravity load that changes with angle. |

**MotionMagic profile:**

| Parameter | Value |
|---|---|
| CruiseVelocity | 2.0 RPS |
| Acceleration | 4.0 RPS² |
| Jerk | 40.0 |

### PID — Roller (Slot 0 — VelocityVoltage)

| Gain | Value |
|---|---|
| kS | 0.15 |
| kV | 0.12 |
| kP | 0.3 |

Same gains as indexers — simple velocity control.

### Associated commands
- **IntakeOutCommand**: Deploys pivot to 0.653 rot + starts roller at 55 RPS (instant — finishes immediately, MotionMagic moves in background)
- **IntakeInCommand**: Stows pivot to 0.3 rot + stops roller (instant)
- **IntakeIrrigateCommand**: Oscillates pivot ±0.02 rot around stowed position at 2.5 Hz to dislodge stuck balls
- **Operator A**: Toggles between deployed and stowed

## 7. Vision

### What it does
Manages PhotonVision cameras for AprilTag detection. Currently only the turret-mounted camera is active. Provides:
- Turret camera yaw (tx) — how far off-center the target is
- Distance to hub — corrected for camera offset
- Diagnostic telemetry (latency, distance components)

### Hardware
- **Turret camera**: USB camera on coprocessor running PhotonVision (name: "mugilanr")
- Right and left cameras exist but are **disabled** (`ENABLE_RIGHT_CAM = false`, `ENABLE_LEFT_CAM = false`)

### Turret camera transform
The turret camera moves with the turret, so its transform from robot center is recalculated each cycle:
```
Robot Center → Turret Pivot → [Turret Rotation] → Camera
```
- `ROBOT_TO_TURRET_PIVOT`: (-5.5", +6.5", 18.064") with 180° yaw (turret faces backward at position 0)
- `TURRET_PIVOT_TO_CAM`: (0", 5.25", 8") with -15° pitch (tilted up)
- The turret rotation is dynamically inserted between these two transforms

### Distance calculation
```
1. Get 3D camera-to-tag translation from PhotonVision (X, Y, Z in camera frame)
2. Extract horizontal distance: sqrt(X² + Y²)  — ignores height difference
3. Add CAMERA_TO_BUMPER_OFFSET_METERS (currently 20") to approximate bumper-to-hub distance
```
The interpolation tables were calibrated with bumper-to-hub tape measurements, so the camera distance must be converted to match.

### Tag filtering
Only **driver-station-facing** hub tags are tracked:
- Red: tags 9, 10 (face toward red driver station)
- Blue: tags 25, 26 (face toward blue driver station)
- `BENCH_TEST_ANY_TAG = false` — set to `true` for bench testing with any tag

### Telemetry fields
| Field | Description |
|---|---|
| `rawCameraDist3d` | Raw 3D norm of camera-to-tag vector |
| `horizontalCameraDist` | Horizontal-only distance (Z removed) |
| `correctedBumperDist` | Horizontal + bumper offset (what interp tables use) |
| `cameraToTargetX/Y/Z` | Individual components of camera-to-tag vector |
| `visionLatencyMs` | PhotonVision pipeline processing time |
| `timeSinceLastFrameMs` | Total time since last fresh frame (includes network delay) |

## 8. Climb

### What it does
Extends/retracts a climbing mechanism to reach cage rungs. Leader-follower motor configuration.

### Hardware
- **Leader motor**: TalonFX (CAN ID 40)
- **Follower motor**: TalonFX (CAN ID 41) — follows leader in **opposed** direction
- **Encoder**: CANcoder (CAN ID 67)
- **CAN Bus**: "mechanisms"

### Gear ratio
```
Documented GEAR_RATIO = 12.0 (but CLIMB_GEAR_RATIO = 1.0 is used for SensorToMechanismRatio)
```
This is inconsistent — the `GEAR_RATIO` constant exists but isn't used in the config. The climb uses a remote CANcoder with `SensorToMechanismRatio = 1.0`, meaning positions are in CANcoder rotations.

### PID (Slot 0 — MotionMagic)

| Gain | Value | Purpose |
|---|---|---|
| kP | 0.1 | Low — climb is slow and controlled |
| kG | 0.5 | **Gravity compensation** — fights the robot's weight. Uses `Elevator_Static` gravity type. |
| Manual kG | 0.5 | Added as voltage offset during manual control |

### Status
**Currently disabled** (`climb = null` in RobotContainer). The climb subsystem code exists but is not instantiated.

### Associated commands
- **LevelCommand**: Two-phase climb to a specific level (inner position, then outer position)
- **Sequence**: Sequential command group that chains LevelCommands for multi-level climbs
- **ManualClimbCommand**: Direct voltage control from joystick (±12V + gravity compensation)
